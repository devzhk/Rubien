# PDF asset sync (B8) — design

**Status:** drafted 2026-04-29; awaiting user review before implementation plan.
**Scope:** ship cross-device PDF binaries by adding a `CDReferencePDF` sibling CKRecord carrying a `CKAsset`, an on-demand download path for resource-constrained clients, LRU cache eviction, and a one-shot backfill of the developer's existing local PDFs. Establishes an architectural invariant — *no synced table holds a local-only column* — so the class of bug we hit on 2026-04-29 (silent `pdfPath` blanking on every pull) cannot recur.

## Context

`RubienSync` ships scalar metadata across CloudKit: Reference, Tag, ReferenceTag, PDFAnnotation, WebAnnotation, MetadataIntake, MetadataEvidence, PropertyDefinition, PropertyValue, DatabaseView. PDF *binaries* are explicitly device-local — `pdfPath` on Reference is a filename inside this device's Group Container `PDFs/` directory, generated by `PDFService.importPDF` with a UUID prefix. The original sync rollout deferred file sync as a follow-up commit ("CKAsset / PDF file sync — separate commit in this work stream") and the source carries forward-references to that work as B8 (`SyncedLibrary.swift:15`, `SyncConstants.swift:42`).

**The bug B8 must learn from.** On 2026-04-29 a routine launch of the signed `Rubien.app` blanked `pdfPath` on every Reference. Root cause in `Sources/RubienSync/SyncEntityDispatch.swift`: the apply path decoded a CKRecord into a fresh `Reference(record:)` (which sets `pdfPath = nil` because `pdfPath` isn't a CKRecord field by design), then called `row.update(db)`, which is a full-row write — wiping the local-only column on every pull. The structural property the code didn't honor: *when a CKRecord schema is a strict subset of a row's persisted columns, the apply path must preserve the unsynced columns*. We fixed it for `pdfPath` specifically, but the shape of the bug — local-only columns living on synced tables — is a permanent footgun. Every column we'd naively add to support B8 (cache state, asset version, content hash, last-opened) is another instance.

**iOS context.** The user has confirmed iPad/iPhone is on the next-six-months roadmap. iOS clients are resource-constrained: a Mac can mirror the entire library happily, an iPhone cannot. B8 must therefore design for *on-demand* download from day one — not because today's user runs an iPad, but because retrofitting on-demand semantics onto a "mirror everything always" design is a much larger refactor than building it correctly the first time.

## Scope

**In scope:**

- `CDReferencePDF` CKRecord type carrying a `CKAsset` plus content hash, asset version, original filename, and modification timestamp.
- New local-only DB tables: `pdfCache` (per-device materialization state) and `pdfUploadQueue` (per-device "pending push" tracker). Neither registered with `SyncEntityType`; neither observed by the dirty-tracking triggers; neither has a CKRecord.
- v2 schema migration: drop `reference.pdfPath`, copy existing values into `pdfCache` + `pdfUploadQueue` rows.
- Push path (Mac-side) — drain `pdfUploadQueue`, attach files as `CKAsset`, push.
- Pull path — apply `CDReferencePDF`, write asset to local `PDFs/`, upsert `pdfCache`. Mac materializes eagerly; iOS receives the record but defers the actual file fetch to user-tap.
- On-demand download UX: a `.pdfAvailability(refId)` view modifier with states `.materialized(URL)`, `.downloading(progress)`, `.error(retry)`, `.notUploaded`. Reader integration. Detail-panel still shows annotation count when not materialized.
- LRU cache eviction with a hidden 5GB default cap. Eviction skips rows still in `pdfUploadQueue` (must not lose unsynced work).
- One-shot backfill of the developer's 31 existing local PDFs on first launch after B8 ships.
- Schema-invariant test that fails CI when any synced table grows a column without a matching CKRecord field.
- Cleanup of pre-existing `dateModified`-not-in-CKRecord drift across the other record types so the invariant test allow-list can stay empty.
- `rubien-cli pdf status <id>` and `pdfBackfillRemaining` field on `rubien-cli sync status`.

**Out of scope (Open follow-ups, not blocking this spec):**

- "Make available offline" pin toggle (the (c) option from brainstorming) — defer; layer onto `pdfCache` as one more column.
- User-configurable cache cap — surface a Settings slider once we have telemetry on real-world usage.
- "Sync over Cellular" toggle for iOS — once iPad ships, default `discretionary=true` on assets so CKSyncEngine waits for Wi-Fi; settings UI to override comes later.
- Multi-asset per Reference (supplementary materials, slides) — current model is 1:1; expanding is a feature scope decision, not a sync concern.
- Content-hash-based asset dedup across multiple References pointing at the same paper — the model assumes 1:1 ref↔asset for v1.
- Annotation drift across diverging PDF versions on different devices — niche; mitigation (warn on hash mismatch) is documented but not built.
- A-pks UUID migration — orthogonal; same status as in the original sync spec.

## Design choices

### Architectural invariant: synced tables hold no local-only columns

This is the structural lesson from the 2026-04-29 bug. Every column on a table that participates in CloudKit sync must have a corresponding field on that entity's CKRecord schema. Per-device state — cache materialization, last-opened timestamps, retry counters, pending-upload markers — lives in dedicated DB tables that:

1. Are not enumerated in `SyncEntityType` (so `applyRemoteRecord` and `buildPushRecord` will never touch them).
2. Are not registered in `AppDatabase.syncedTables` (so dirty-tracking triggers don't fire on writes).
3. Have no `*Record.swift` file in `RubienSync/`.

A new test (`Tests/RubienSyncTests/SyncSchemaInvariantTests.swift`) introspects each `SyncEntityType` case at runtime: it opens a fresh `AppDatabase`, lists the table's columns via `pragma_table_info`, retrieves the entity's CKRecord field names via a new `static var allFieldNames: [String]` on each record extension (a one-line addition next to the existing `RecordField` enum — Mirror can't see static lets inside caseless enums, so we publish the list explicitly), and asserts every persisted column has a corresponding RecordField. An explicit allow-list documents intentional exceptions (none expected after C5 cleanup; the allow-list is meant to start empty and stay empty). Drift fails the test, not the user's data.

### Sibling CKRecord, not asset-on-Reference

`CDReferencePDF` is a separate record type, 1:1 with Reference, recordName `"referencePDF:<refId>"`. Reasoning:

1. **CKRecord change granularity.** CloudKit treats the CKRecord as the unit of change. Putting the CKAsset on `CDReference` itself would mean every scalar edit (changing reading status, adding a tag, editing notes) reuploads the PDF asset — catastrophically inefficient.
2. **Independent dirty tracking.** Asset state (uploaded? failed? hash mismatched?) is independent of metadata state. Decoupling them lets either evolve without dragging the other.
3. **Forward compatibility.** When we eventually want multi-asset per Reference (supplementary materials), CDReferencePDF generalizes to multiple records sharing a `referenceId` field; CDReference doesn't have to know.

The 1:1 mapping is enforced by recordName convention only — the local schema lets `pdfCache` hold at most one row per `referenceId` via PK. If multi-asset becomes a feature, the recordName scheme generalizes (`"referencePDF:<refId>:<assetIdx>"`) without record-type changes.

### Cache lives in `pdfCache`, not on Reference

`pdfCache(referenceId PRIMARY KEY, localFilename TEXT NOT NULL, contentHash TEXT NOT NULL, assetVersion INTEGER NOT NULL, materializedAt DATETIME NOT NULL, lastOpenedAt DATETIME NOT NULL)` is a local-only table. `Reference.pdfPath` goes away. The lookup helper `AppDatabase.pdfFilename(for: refId)` joins through `pdfCache`. Foreign key from `pdfCache.referenceId` to `reference.id` with `ON DELETE CASCADE` so deleting a Reference correctly removes its cache row.

This is the single biggest concrete win against the bug class: post-B8, the `reference` table has zero columns that aren't in `CDReference`'s CKRecord schema, and the schema-invariant test will keep it that way.

### Upload queue lives in `pdfUploadQueue`, not on Reference

`pdfUploadQueue(referenceId PRIMARY KEY, localFilename TEXT NOT NULL, queuedAt DATETIME NOT NULL)`. Local-only. Drained by a `PDFUploadQueue` actor running off the SyncedLibrary actor's lifecycle. An entry exists from the moment a local PDF is registered (import or backfill) until the engine confirms the push succeeded — at which point the row is removed and the cached server systemFields are stored alongside the cache row for future updates.

### Eviction must skip files still in the upload queue

The single non-obvious eviction rule. Without it, an iPad import of a 50MB PDF that pushes the cache past cap could trigger LRU eviction of the just-imported file (its `lastOpenedAt` is fresh, but if cap is exactly hit other "less recently opened" files might be picked first; an aggressive cap could evict the new one too if the cache was already very full). Either way, "evict a file that hasn't been pushed to cloud yet" loses unsynced work irrecoverably.

The invariant: `evictIfOverCap` excludes any `referenceId` present in `pdfUploadQueue`. Belt-and-suspenders: the upload-queue drainer handles "file vanished" by removing the queue entry and logging — so even if eviction somehow violates the rule, sync degrades to "push didn't happen" rather than "push tried with a missing file and surfaced as a sync error."

### On-demand download UX (resource-constrained clients only)

Mac mirrors the library: on pull, applyRemoteRecord materializes the asset immediately. The user expects to be able to reach for any PDF without thinking about it.

iOS does not mirror. Pull receives the lightweight `CDReferencePDF` *record* (the asset is downloaded lazily by CloudKit because we declare it as such), populates `pdfCache` with metadata only (filename + hash + version + materializedAt = NULL), and waits. When the user taps the reference, the reader consults `pdfCache`: if `materializedAt` is NULL, it transitions the view modifier to `.downloading(progress)`, asks `SyncedLibrary.fetchAsset(refId:)` to materialize, then renders.

The Mac/iOS decision is made at compile-time via `#if os(iOS)` in the apply path — no runtime device-class detection.

### Eager backfill of existing local PDFs

When B8 ships, the v2 migration walks every Reference row that has a `pdfPath` value and the file actually exists in `PDFs/`, copies the value into `pdfCache` (`materializedAt = now`, `contentHash = sha256(file)`), and inserts a `pdfUploadQueue` entry. On the first post-migration launch with sync enabled, the queue drainer hashes and uploads them all. CKSyncEngine rate-limits, so eager doesn't mean a thundering herd. The Settings sync section shows a small "Uploading X PDFs to iCloud (Y of Z complete)" indicator while the queue drains; the indicator disappears when the queue empties.

### Default cache cap: 5GB, hidden

For v1 the cap is a `let kDefaultCacheCap: Int64 = 5_000_000_000` in `PDFAssetCache`. No Settings UI. Once we have real-world data on cache hit rate and user complaints, expose a slider (Open follow-up). The number is a guess — research-tool users typically have low-hundreds of PDFs at low-tens of MB each, so 5GB covers the bulk on iPad without dominating phone storage.

### Conflict policy on assets

If two devices upload different bytes for the same Reference within the same sync window, last-write-wins by CKSyncEngine's default `.ifServerRecordUnchanged` semantics — the second pusher's write fails, the engine retries with the latest server record's systemFields, and overwrites. The losing device's local file is replaced on next pull. Existing annotations on the losing device may anchor to outdated coordinates — surfaced as an Open follow-up under "annotation drift," not handled in v1.

### `dateModified`-on-record cleanup (folded into B8)

Audit found `dateModified` is currently a DB column on five synced tables (`tag`, `pdfAnnotation`, `webAnnotation`, `propertyDefinition`, `propertyValue`) but absent from their CKRecord schemas. The mismatch hasn't caused user-visible damage because `dateModified` is auto-stamped on every write, so the post-pull "missing → stamped fresh" outcome is correct-enough. But it would defeat the schema-invariant test. Two options: (a) add `dateModified` to those CKRecord schemas, (b) accept it as the only entry in the invariant allow-list.

C5 picks (a). The cost is minor (5 extra fields), and an empty allow-list is a stronger invariant than a one-entry allow-list (no judgment calls about what's "auto-stamped enough" to be exempt). Concretely: extend each affected `*Record.swift` to include `dateModified` in `populate`/`init`/`RecordField`. Pre-existing rows on the cloud will have nil for that field; decoders fall back to `Date()` in their `init?` (already the pattern for missing optional fields). No data migration required.

## Architecture

```
+----------------------------------+        +---------------------------------+
|             iCloud               |        |      Local: ~/Library/Group     |
|  Private CKDatabase, libraryZone |        |    Containers/.../Rubien/       |
|                                  |        |                                 |
|  CDReference (scalar metadata)   |  push  |  library.sqlite                 |
|  CDTag, CDPDFAnnotation, ...     | <----> |    reference (no pdfPath)       |
|                                  |  pull  |    pdfCache (local only)        |
|  CDReferencePDF (CKAsset blob)   |        |    pdfUploadQueue (local only)  |
|                                  |        |                                 |
+----------------------------------+        |  PDFs/                          |
              ^                             |    <uuid>_<file>.pdf            |
              |                             |                                 |
              | drives                      +---------------------------------+
              v
   +---------------------+
   | SyncedLibrary actor |
   |                     |       +-------------------------+
   |  CKSyncEngine       |       | PDFAssetCache (actor)   |
   |  + delegate methods | <---> |  materialize / pathFor /|
   |                     |       |  evictIfOverCap /       |
   |  + asset push hook  |       |  markOpened             |
   |  + asset pull hook  |       +-------------------------+
   +---------------------+                ^
              ^                           | reads/writes
              | drains                    |

   | PDFUploadQueue      |       | Reader UI               |

   |  - attach as asset  |       |   .downloading(p)       |
```

## Components

### New types in `Sources/RubienSync/`

- `ReferencePDFRecord.swift` — `populate(record:)`, `init(record:)`, `makeRecord(recordName:referenceId:fileURL:hash:version:filename:)`. Mirrors the existing per-entity record file pattern. `RecordField` enum keys the field names. The asset is attached as `CKAsset(fileURL:)`; `CKAsset` requires a real file URL (it stages and uploads in the background).
- `SyncEntityType.referencePDF` case in `SyncEntityDispatch.swift`. `buildPushRecord` reads the `pdfCache` row + the local file, computes/refreshes content hash, builds the record. `applyRemoteRecord` materializes the asset (Mac) or just records metadata (iOS) and upserts the `pdfCache` row. `applyRemoteDelete` removes the `pdfCache` row and (Mac only) deletes the local file.
- `PDFUploadQueue` actor — drains `pdfUploadQueue` rows by asking the engine to push `CDReferencePDF` records. Owned by `SyncedLibrary`; lives only while sync is enabled. If sync is off, queue rows accumulate harmlessly and drain on next enable. Single concurrent upload (the engine itself batches, no need to pile on); resumes on app launch by re-reading the table.

### New types in `Sources/RubienCore/`

- `PDFAssetCache` actor — fronts the `pdfCache` table. Methods:
  - `materialize(referenceId:assetURL:hash:version:filename:) async throws -> URL` — copies an asset (from CKAsset's `fileURL` or a local import path) into `PDFs/`, upserts the cache row, returns the materialized local URL.
  - `pathFor(referenceId:) async -> URL?` — returns local URL if materialized, else nil.
  - `metadataFor(referenceId:) async -> CacheEntry?` — returns the row including version/hash even if not materialized.
  - `markOpened(referenceId:)` — bumps `lastOpenedAt`. Called by reader.
  - `evictIfOverCap()` — LRU evict, skipping rows in `pdfUploadQueue`.
  - `dematerialize(referenceId:)` — keep the row, remove the file (e.g., explicit user "remove cached PDF").
- v2 migration in `AppDatabase.swift`. New `registerMigration("v2", ...)` block. Creates `pdfCache` and `pdfUploadQueue` tables. Populates them from existing `reference.pdfPath` values + on-disk PDF files. Drops the `pdfPath` column from `reference` (SQLite 3.35+ supports `ALTER TABLE DROP COLUMN` directly; macOS 14's bundled SQLite is well past that, but the implementer can fall back to the standard rebuild-table-without-the-column pattern if any GRDB compatibility wrinkle surfaces). The `dateModified` field additions on other record types (C5) are CKRecord-only changes — no DB schema change required.

### Modifications to `Sources/Rubien/`

- `Reference.pdfPath` Swift property removed. Replace call sites with `db.pdfFilename(for: refId)` (sync) or `cache.pathFor(refId)` (async). Affected files: `ReferenceListView`, `ReferenceDetailView`, `PDFReaderView`, `ReaderWindowManager`, `AddReferenceView`, `ContentView`, `SearchReferenceView`, `PDFOutlineSidebarView`. Most are simple substitutions; `ReferenceDetailView` and `ContentView` have edit flows that need cache writes via `PDFAssetCache.materialize`.
- New view modifier `.pdfAvailability(referenceId:)` returning a SwiftUI binding the reader can switch on.
- Settings → Sync section: cache size used / 5GB cap progress bar, "X PDFs pending upload" indicator while non-zero.

### New CLI

- `rubien-cli pdf status <id>` — JSON output with `cached`, `localFilename`, `assetVersion`, `contentHash`, `materializedAt`, `lastOpenedAt`, `inUploadQueue`.
- `rubien-cli sync status` extended with `pdfBackfillRemaining: Int` (count of `pdfUploadQueue` rows).

## Data flow

### Flow A — push: import PDF on Mac

1. `PDFService.prepareImportedPDF` writes file to `PDFs/<uuid>_<name>.pdf` (existing).
2. Reference row inserted (existing dirty trigger fires for CDReference).
3. **New:** `PDFAssetCache.materialize` upserts `pdfCache` row with hash + materializedAt=now.
4. **New:** `pdfUploadQueue` row inserted.
5. `PDFUploadQueue` drainer notices the new row, hashes (already done at materialize) the file, asks `SyncedLibrary.pushReferencePDF(referenceId:)` to enqueue a `CDReferencePDF` push.
6. Engine uploads. On `didSendChanges` for the record: drainer removes the `pdfUploadQueue` row, stores cached server systemFields keyed by `"referencePDF:<refId>"` for future updates.

### Flow B — pull: Mac receives a PDF that another device pushed

1. Engine delivers a `CDReferencePDF` record event.
2. `SyncEntityType.referencePDF.applyRemoteRecord` runs.
3. CKAsset's `fileURL` is already populated (CloudKit downloaded it).
4. **Mac path:** copy/rename the asset file into local `PDFs/<uuid>_<originalFilename>`, upsert `pdfCache` row with `materializedAt = now`.
5. **iOS path:** *don't* copy the asset. Upsert `pdfCache` row with `materializedAt = NULL`. (We declare the asset as non-eager via `CKRecord.shouldFetchAssets = false` if available, or rely on lazy CKAsset fetch.)

### Flow C — open on iOS, cache miss

1. User taps reference.
2. Reader checks `cache.pathFor(refId)` → nil because `materializedAt = NULL`.
3. View modifier transitions to `.downloading(progress: 0)`.
4. Reader calls `SyncedLibrary.fetchAsset(refId:)` which asks the engine to fetch the asset for record `"referencePDF:<refId>"`.
5. On completion, applyRemoteRecord runs again with the asset materialized; cache row updated with `materializedAt = now`.
6. View modifier transitions to `.materialized(URL)`; reader renders.

### Flow D — push: import PDF on iPad

Symmetric to Flow A. iPad's local `PDFs/` directory acquires the file via `PDFService.prepareImportedPDF`; `pdfCache` and `pdfUploadQueue` rows created on iPad; drainer pushes; Mac pulls and materializes per Flow B Mac path. iPad's freshly-imported PDF is `materializedAt = now` from the moment of import.

### Flow E — eviction on iOS under cache pressure

1. `PDFAssetCache.materialize` finishes a write; checks total cache size.
2. If over cap: select rows ordered by `lastOpenedAt` ascending, *excluding* `referenceId` present in `pdfUploadQueue`.
3. Delete files for the oldest until under cap, set those rows' `materializedAt = NULL` (preserve metadata so a future tap can re-fetch).
4. If everything except upload-queue rows is already evicted and we're still over cap: log a warning, accept the overage. Cap is a soft limit; never break the don't-evict-unsynced invariant.

### Flow F — first-launch backfill after B8 ships

1. v2 migration populates `pdfCache` and `pdfUploadQueue` from existing `reference.pdfPath` + on-disk files.
2. App launches, sync enabled, `SyncedLibrary` starts.
3. `PDFUploadQueue` drainer reads rows ordered by `queuedAt`; drains.
4. Settings shows "Uploading 31 PDFs to iCloud (4 of 31 complete)" until empty.
5. Pre-existing CKSyncEngine retry/throttle handles failures; nothing app-specific.

### Flow G — recovery: local PDF file vanishes (user nuked `~/Library`)

1. Reader opens; `cache.pathFor(refId)` returns a URL; the actual file doesn't exist (`FileManager.fileExists`).
2. Reader treats this identically to "cache miss": transition to `.downloading`, fetch asset.
3. If the asset isn't in cloud either (e.g., this PDF was added recently and the queue hadn't drained when the user nuked): view modifier shows `.notUploaded` with a "Re-attach PDF" affordance.

## Error handling

- **Upload failure (transient):** CKSyncEngine retries automatically per its existing policy. After engine gives up (per-record limit), the existing "Sync Issues" banner surfaces with the failed record; user can manually retry by toggling sync off/on or via a future "retry sync" CLI subcommand.
- **Download failure (transient):** same. Reader's `.error(retry)` state offers an explicit retry button that re-asks the engine.
- **Hash mismatch on download:** local `contentHash` doesn't equal the downloaded asset's SHA-256. Re-fetch once. On second mismatch, log and surface as `.error("PDF integrity check failed")`. Do not silently overwrite — the right move is investigation, not paper-over.
- **Local file present but cache row missing (drift):** treat the file as orphaned. Do not auto-delete (could be user data). Log; offer a future CLI command to reconcile (Open follow-up).
- **iCloud quota exceeded:** existing path from `make-sync-runnable` spec applies (Flow D in that spec). Backfill stalls until quota frees up; user sees the existing quota banner.
- **CKAsset upload size limit:** CloudKit caps assets at ~250 MB. PDFs in research are ~5–50 MB; well under the limit. If a PDF exceeds (e.g., a large dataset PDF), fail with `.error("PDF too large to sync")` and leave the file local-only. Belt-and-suspenders: pre-flight size check before queuing.
- **Eviction race with upload:** eviction explicitly excludes upload-queue rows (see Design choices). Plus the drainer handles "file vanished" gracefully — removes the queue row, logs, no crash.

## Testing

### Unit tests (no CloudKit, no entitlement)

- `ReferencePDFRecordTests` — round-trip a `pdfCache` row + file → CKRecord → DB; assert all fields preserved including hash, version, originalFilename, dateModified.
- `SyncEntityDispatchTests` extension — `applyRemoteRecord(.referencePDF)` test: seed an in-memory DB with a Reference, build a CDReferencePDF with a temp-file CKAsset, apply, assert `pdfCache` row created and file copied.
- **`SyncSchemaInvariantTests`** — for each `SyncEntityType` case: open a fresh AppDatabase, list `pragma_table_info(<table>)` columns, list the entity's `RecordField` static-let strings, assert every column except those in an explicitly-empty allow-list has a matching RecordField. **This is the bug-class defense; failure should block CI.**
- `PDFAssetCacheTests` — materialize, pathFor, markOpened, eviction (with and without upload-queue rows present), dematerialize.
- `PDFUploadQueueTests` — enqueue, drain a single row, drain handles missing-file gracefully.
- `MigrationV2Tests` — seed a v1 DB with Reference rows and a stub `pdfPath` column, write fake PDFs to the storage dir, run the migrator, assert: pdfPath column dropped, pdfCache rows present with hashes, pdfUploadQueue rows present, file count unchanged on disk.

### CLI tests

- `pdf status <id>` JSON shape stable across builds.
- `sync status` JSON includes `pdfBackfillRemaining` and decreases as the queue drains in a manual smoke test.

### What we can't test automatically

- The actual CKAsset roundtrip through CloudKit. Same as scalar tests in the existing spec — manual smoke test on two devices is the only path.
- iOS-specific flows. Until the iPad target lands, we test the iOS-conditional code paths via build-time `#if os(iOS)` matrices and unit tests run under the iOS simulator (added to the test plan).

### Manual smoke test (after C2 ships)

1. Mac A: import a 10MB PDF, observe upload completes (Settings shows "0 PDFs uploading" within ~30s on a normal connection).
2. Mac B: pull, observe the PDF appears in `~/Library/Group Containers/.../Rubien/PDFs/` and the reference renders on tap.
3. Mac A: edit the reference notes; verify the asset is *not* re-uploaded (CKRecord change unit is per-record, not per-row; CDReference and CDReferencePDF are independent).
4. Mac B: delete the reference; verify both `CDReference` and `CDReferencePDF` records on Mac A are deleted (FK cascade locally + tombstone propagation across).

## Commit decomposition

Six commits, each independently shippable and reverible. Feature flag (`pdfAssetSyncEnabled`) gates the runtime push/pull paths until C5; up to that point B8 is dormant code that ships with the binary but doesn't run.

1. **C1 — Schema + local cache plumbing.** v2 migration creates `pdfCache` and `pdfUploadQueue`, copies existing `pdfPath` data, drops the column. `PDFAssetCache` actor implemented. App code switched from `Reference.pdfPath` to cache lookups. No CloudKit changes. Tested via `MigrationV2Tests` and `PDFAssetCacheTests`. Mergeable on its own; no behavior change visible to user.

2. **C2 — Push path.** `ReferencePDFRecord.swift`. `SyncEntityType.referencePDF` case (push only). `PDFUploadQueue` actor. Backfill path enqueues existing local PDFs. Behind `pdfAssetSyncEnabled` feature flag (off by default). Round-trip tests + `SyncEntityDispatchTests`-extension cover the new dispatch case.

3. **C3 — Pull path.** Apply-remote for `.referencePDF`: Mac materializes; iOS records metadata only. `.pdfAvailability` view modifier. Reader integration. On-demand fetch via `SyncedLibrary.fetchAsset(refId:)`. Manual smoke test on two Macs is now possible.

4. **C4 — Eviction + cache cap.** `PDFAssetCache.evictIfOverCap` with skip-upload-queue invariant. Settings sync section shows cache used + backfill progress. `pdfBackfillRemaining` added to `sync status` JSON. `rubien-cli pdf status` subcommand.

5. **C5 — Schema invariant test + dateModified cleanup.** `Tests/RubienSyncTests/SyncSchemaInvariantTests.swift`. Add `dateModified` field to `TagRecord`, `PDFAnnotationRecord+CloudKit`, `WebAnnotationRecord+CloudKit`, `PropertyDefinitionRecord`, `PropertyValueRecord`. Allow-list in the invariant test stays empty. **Flip `pdfAssetSyncEnabled` to true** as part of this commit so all tests pass with the flag on.

6. **C6 — Backfill rollout polish.** Settings UX for "Uploading X PDFs to iCloud" indicator. CLI extensions verified end-to-end. Documentation updates to `CLAUDE.md` (sync section gains a "PDF asset sync" subsection) and `Docs/Sync-Runbook.md` (manual smoke test + diagnostics for asset-specific failures).

## Open follow-ups (tracked, not blocking this spec)

- **(c) Pin-for-offline.** Add `isPinned BOOLEAN NOT NULL DEFAULT 0` column to `pdfCache`; eviction excludes pinned rows. UI: per-reference "Make available offline" toggle. ~half-day.
- **User-configurable cache cap.** Settings → Sync → "Cache up to X GB of PDFs" slider. Default stays 5GB. Add `userCacheCapBytes` to `RubienPreferences`; `PDFAssetCache` reads it.
- **iOS cellular policy.** When iPad target lands, default `discretionary=true` on CKAsset uploads. Add a "Sync over Cellular" toggle in Settings.
- **Multi-asset per Reference.** Generalize recordName scheme to `"referencePDF:<refId>:<assetIdx>"`. Schema change: `pdfCache` PK becomes (referenceId, assetIdx). Reader UI: "Attach supplementary PDF" affordance.
- **Content-hash-based dedup across References.** Two refs pointing at the same paper share a single CKAsset. Save iCloud quota; complicate identity. Probably needs a separate `pdfAsset(hash, ...)` table independent of references.
- **Annotation drift across diverging PDF versions.** When `assetVersion` increments and content hash changes, surface a warning if any annotations exist for the old version. Mitigation could be: keep the old file around for annotation rendering, mark it `legacy`, prompt user to migrate.
- **Recover-from-orphan-files CLI command.** `rubien-cli pdf reconcile` — finds files in `PDFs/` with no matching `pdfCache` row (or vice versa), reports, optionally rebinds.
